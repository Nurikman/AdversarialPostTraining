# Local MLX experiment

A parallel pipeline to `openai_sft_wrong_reasoning_experiment/` that replaces OpenAI's managed fine-tuning + Responses API with **local** SFT + inference on Apple Silicon using [MLX-LM](https://github.com/ml-explore/mlx-examples/tree/main/llms).

Why bother:

- **Reproducibility** — anyone can re-run the entire experiment from this repo. The OpenAI version isn't really reproducible because `ft:gpt-4.1-nano-2025-04-14:personal::DaN40hmK` is an opaque per-account artifact.
- **Cost / rate limits** — none.
- **Ablation surface** — full SFT vs LoRA, layer freezing, learning rate sweeps, gradient analysis, all impossible via OpenAI's API.
- **Generalization** — if the "wrong-reasoning damages reasoning" finding only shows up on `gpt-4.1-nano` and not on any open model, that's a fact about gpt-4.1-nano, not about post-training.

The eval set, metrics, CSV schema, and plotting code are **shared** with the OpenAI pipeline — `plot_correct_rate.py` reads from either or both.

## Setup

```bash
pip3 install --user mlx-lm pandas
export PATH="$HOME/.local/bin:$PATH"   # only if mlx_lm.lora isn't on PATH
```

Apple Silicon required (M-series). Linux/CUDA users should rewrite `train_mlx.py` and `eval_local.py` against `transformers` + `peft` + `trl`.

## Files

| file | what it does |
|---|---|
| `convert_sft_to_mlx.py` | Copies the 7 `sft_*.jsonl` files into `mlx_data/<condition>/{train,valid}.jsonl`. Validation set is *always* `sft_baseline_clean_100` so valid loss is a direct measure of "can the model still produce correct reasoning?" |
| `train_mlx.py` | Wraps `mlx_lm.lora` to fine-tune one or all conditions, with `--seeds N` matching the OpenAI launcher's interface. Defaults to LoRA (fast), `--fine-tune-type full` for OpenAI-equivalent full SFT. |
| `eval_local.py` | MLX inference mirror of `eval_models.py`. Same CSV schema (Wilson CI, wrong_adoption split, poisoned/clean breakdown) so downstream `plot_correct_rate.py` works unchanged. |
| `run_all_local.sh` | End-to-end orchestrator: convert → train all conditions → eval all conditions → plot. |
| `mlx_data/` | Generated by `convert_sft_to_mlx.py`. Gitignored. |
| `adapters/` | Generated by `train_mlx.py`. One subdir per `(condition, seed)`. Gitignored. |
| `eval_results_*.csv` | Generated by `eval_local.py`. Same schema as OpenAI pipeline. Gitignored. |

## Default model: Qwen2.5-Math-1.5B-Instruct

Reasoning:

- Strong baseline GSM8K (~84% on the published card) — there's something to degrade.
- 1.5B fits comfortably on M-series with batch_size ≥ 4 in full SFT.
- Permissive license.
- First model load is ~3 GB download (one-time).

Alternatives via `--model`:

| model | size | reason to pick it |
|---|---|---|
| `Qwen/Qwen2.5-Math-1.5B-Instruct` | 1.5 B | default — strong math at small size |
| `Qwen/Qwen2.5-1.5B-Instruct` | 1.5 B | general-purpose, weaker base GSM8K (~73%) |
| `Qwen/Qwen2.5-Math-7B-Instruct` | 7 B | strongest math at this scale; ~3× slower |
| `Qwen/Qwen2.5-3B-Instruct` | 3 B | middle ground |

## End-to-end run

```bash
# Defaults: Qwen2.5-Math-1.5B-Instruct, 1 seed, LoRA, 100 iters
./run_all_local.sh

# 3 seeds, full SFT (closer match to OpenAI's behavior), more iterations
SEEDS=3 FINE_TUNE_TYPE=full ITERS=200 ./run_all_local.sh

# Bigger model
MODEL=Qwen/Qwen2.5-Math-7B-Instruct ./run_all_local.sh
```

Estimated runtime on M4 Pro (Qwen2.5-Math-1.5B-Instruct, LoRA, 1 seed):
- convert: <1 s
- train (7 conditions × 100 iters): ~10–15 min total
- eval (7 conditions × 2 models × 100 items): ~25–40 min total
- **end-to-end: ~45–60 min for the full ladder**

For full SFT with 3 seeds: roughly 3–4 h.

## Step-by-step (manual)

```bash
# 1. One-time: structure the data
python3 convert_sft_to_mlx.py

# 2. Train one condition with 3 seeds
python3 train_mlx.py --condition underwrite_010 --seeds 3

# 3. Eval base + the 3 fine-tunes
python3 eval_local.py \
    --condition underwrite_010 \
    --base-model Qwen/Qwen2.5-Math-1.5B-Instruct \
    --models Qwen/Qwen2.5-Math-1.5B-Instruct \
             adapters/underwrite_010/seed_1 \
             adapters/underwrite_010/seed_2 \
             adapters/underwrite_010/seed_3

# 4. (Optional) OOD eval on GSM-Symbolic
python3 ../openai_sft_wrong_reasoning_experiment/fetch_gsm_symbolic.py \
    --variant main --per-template 2 \
    --out eval_gsm_symbolic_main.jsonl
python3 eval_local.py \
    --condition ood \
    --base-model Qwen/Qwen2.5-Math-1.5B-Instruct \
    --models Qwen/Qwen2.5-Math-1.5B-Instruct \
             adapters/underwrite_010/seed_1 \
             adapters/underwrite_010/seed_2 \
             adapters/underwrite_010/seed_3 \
    --eval-file eval_gsm_symbolic_main.jsonl

# 5. Plot (the OpenAI experiment's plotter)
python3 ../openai_sft_wrong_reasoning_experiment/plot_correct_rate.py \
    --files eval_results_*_summary.csv \
    --out plots/correct_rate_by_condition_local.html
```

## What to look for

The interesting result lives in three places:

1. **Train loss curves vs valid loss curves during fine-tuning.** Train loss drops on every condition (the model is happily learning whatever you show it). The question is whether valid loss — measured against `sft_baseline_clean_100`, the clean-reasoning control — drops alongside it (training is general improvement) or rises (training is corrupting reasoning). MLX-LM logs both every 25 iters by default.
2. **Per-condition `correct_rate` on `eval_attached_100_cases.jsonl`.** Same comparison the OpenAI experiment does. If the local pipeline replicates "overwrite craters, underwrite_010 degrades by ~10pp with ~0 direct adoption," the finding is real.
3. **Per-template breakdown on GSM-Symbolic.** Concentrated drop on some templates → pattern-specific damage. Uniform drop → broad degradation.

The `baseline` (clean SFT) condition is critical here. A model fine-tuned on 100 clean correct solutions still loses ~5–15 pp on GSM8K-style evals just because any SFT distribution-shifts the model. Without that control, all the underwrite_* "damage" measurements are confounded with normal SFT drift.
